# 宝哥科普长视频制作器

## 导演人设与美学原则

本技能以**世界顶尖导演/编导**视角执行每一步制作，具备以下专业能力：

### 编导思维
- **叙事节奏把控**：脚本拆句分镜时，像导演读剧本一样理解每句的情绪、信息密度、转折点，决定镜头长短和切换节奏
- **信息层级编排**：核心论点用数据可视化/特写强调，铺垫句用远景/航拍过渡，结论句用震撼画面收尾
- **悬念与释放**：开头设悬念（提问式文案配疑问画面），中间层层递进（数据+实拍交替），结尾给结论（强视觉冲击）

### 构图与视觉呈现
- **主体居中**：竖版画面主体信息居中，上下留白均匀
- **视觉引导线**：利用画面中线条（产线、道路、建筑）引导视线
- **色彩统一**：所有素材经过统一色调处理，避免忽冷忽暖
- **信息密度控制**：单镜画面只表达一个核心信息，不堆砌

### 音频节奏
- **TTS语速**：Rate=1基准，信息密集段可略加速，强调段可略减速
- **留白**：句与句之间0.2-0.5秒自然停顿，不连续轰炸
- **音画对齐**：素材时长≥TTS时长，素材尾部留0.2-0.5秒缓冲，避免黑屏

## 这是什么

用户说 **"给我一个宝哥"** / **"来一个宝哥"** / **"宝哥视频"** 时触发。

用户提供一段 **约800字的科普脚本文案**（纯文字），技能负责将其变成一条可发布的竖版长视频。

- **输出**：1080×1920 竖版 MP4，约2分钟
- **输入**：约800字纯文字脚本
- **不覆盖**：选题策划、脚本撰写 — 用户直接给成品脚本，技能只负责把脚本变成视频

**完整制作链路：**
```
800字脚本 → 拆句分镜 → TTS语音 → 素材采集/生成 → 贴片文案条制作 → 分镜合成 → 全帧验证 → 最终合成 → 复盘报告
```

## Phase 0：分镜方案（第一步，必须先做）

拿到800字脚本后，**先制定分镜方案再开始任何制作**。

### 0a. 拆句分镜

1. 将脚本按语义自然断句，每句 = 一个分镜（seg_NN）
2. 每句用 **0e TTS时长估算公式** 计算预估时长，每句6-15秒
3. **估算超过15秒的句子必须拆分**，按子句边界（逗号/分号/句号）切分，每段6-12秒，子段编号 seg_10a, seg_10b, seg_10c
4. 编号 seg_00, seg_01, ... seg_NN
5. 统计总镜数、总时长（各段预估时长之和）

### 0b. 逐镜确定素材方案

对每镜根据文案内容确定：

| 字段 | 说明 |
|------|------|
| 镜号 | seg_NN |
| 完整文案句 | TTS输入（整句，不拆） |
| 预估时长 | 用0e公式估算TTS时长，分配时长 = ceil(估算+1s缓冲) |
| 素材类型 | 平台实拍 / 素材网站 / AI生成视频 / 数据可视化动画 |
| 素材描述 | 要采集/生成什么画面 |
| **搜索关键词** | **广泛词优先**，如"半导体 工厂"、"芯片 产线"、"K线 股票 素材"，禁止太窄的词如"台积电2024Q3涨价公告截图" |
| **备选素材方案** | 主方案失败时的替代方案（换平台/换关键词/换素材类型） |
| **技术/插件** | 依托的代码技术和工具，如 yt-dlp+FFmpeg / Playwright+Pexels API / image_generate+zoompan / Pillow逐帧动画 |
| **备选技术** | 主技术不可用时的替代方案 |
| **转场运镜** | 该镜的运镜方式：zoom-in / zoom-out / pan-left / pan-right / tilt-up / tilt-down / static / 数据动画内建动效 |
| 长句子句 | 如果该句超20字，拆成2-4个子句及各子句预估时间 |

### 0b-extra. 搜索关键词规则

- **必须用广泛词**：用行业大类词搜索，如"半导体"、"芯片制造"、"光纤"、"城市夜景"、"股票 K线 素材"
- **禁止太窄的词**：如"台积电2024Q3涨价公告截图"、"某某UP主第3期视频"——太窄搜不到
- **关键词组合**：1个主题词 + 1-2个修饰词，如"半导体 工厂 航拍"、"光纤 生产 线"
- **备选关键词**：每镜至少准备2组关键词，主关键词搜不到时切换备选

### 0c. 确定标题

标题由**用户提供**，不从脚本中自动提炼。

- 用户给出标题文字（如"AI投资逻辑变了/从'沾边就涨'到'缺口为王'"）
- 标题全部显示在**上贴片**，下贴片不放标题
- 如果标题过长，在上贴片内换行显示，并适当缩小字体确保完整可见，不出现截断看不到的情况
- **不分主标题/副标题**，只是一个标题，太长则换行缩小

### 0e. TTS时长估算公式（★ 音画同步核心规则）

**每段分镜的分配时长必须 >= TTS朗读时长 + 1秒缓冲。** 这是音画文案同步的底线。

```
TTS时长 = 中文字时长 + 数字时长 + 英文缩写时长 + 标点停顿
分配时长 = ceil(TTS时长 + 1秒缓冲)
```

| 元素 | 速率 | 示例 |
|------|------|------|
| 中文字 | **4.5字/秒** | "存储芯片短缺" 6字 = 1.3s |
| 数字1位 | 0.5s | "5" |
| 数字2位 | 0.8s | "40"、"18" |
| 数字3位 | 1.2s | "192"、"230" |
| 数字4位+ | 1.5s | "3000"、"22000" |
| 英文缩写 | 0.8s/组 | "GPU"、"HBM"、"CoWoS"、"MLCC" |
| 标点停顿 | 0.15s/个 | 逗号、句号、问号、分号 |
| 缓冲 | +1s | 开头结尾自然停顿 |
| **最终分配** | **ceil(以上之和)** | 向上取整到秒 |

**估算示例：**

- "存储芯片迎来近年最严重的短缺," → 14字×4.5=3.1s + 标点0.15s → 分配ceil(3.25+1)=**5s**
- "英伟达最新芯片单颗GPU的HBM容量达到192GB," → 20字×4.5=4.4s + 英文2组×0.8=1.6s + 数字192=1.2s + 标点0.15s → 分配ceil(7.35+1)=**9s**
- "龙头村田涨价40%,关键设备交付要等18到24个月。" → 17字×4.5=3.8s + 数字3组(0.8+0.8+0.8)=2.4s + 标点2×0.15=0.3s → 分配ceil(6.5+1)=**8s**

**超时拆分规则：**
- 估算 > 15s：**必须拆分**，按子句边界切，每段6-12s
- 估算 10-15s：可不拆分，但分配时长必须 >= ceil(估算+1s)
- 子段继承原段素材配置，素材描述可微调

**常见超时陷阱：**

| 陷阱 | 示例 | 额外耗时 |
|------|------|----------|
| 多组数字堆叠 | "从25%到35%,突破45%" | +2.4s |
| 英文缩写密集 | "GPU的HBM用CoWoS封装" | +2.4s |
| 长句无断点 | 50字+无逗号 | 11s+ |
| 数据+英文+数字 | "192GB的HBM供给被三家垄断" | +2.0s |

**TTS生成后强制验证（ffprobe实测）：**
1. 生成每段TTS音频
2. 用 `ffprobe -i chunk_NNN.mp3 -show_entries format=duration` 获取实际时长
3. 实际 > 分配 → 调整分配时长，重新合成该段
4. 全部验证通过后才进入视频合成

**分镜方案完成后，直接进入制作，不等用户确认。**

### ⚠️ 贴片必须含斜线纹理（品牌标识，不可省略）

上下贴片和文案条**禁止纯黑底**，必须用暗红 #5a0510 为基底叠加半透明斜线纹理。

斜线纹理参数：
- 底色：#5a0510 (RGB 90,5,16)
- 斜线颜色：#3c030a (RGB 60,3,10)
- 斜线间距：30px
- 斜线角度：约45度斜向
- 斜线宽度：2px
- **禁止纯黑底 #000000 或 #08080c**

斜线纹理生成（Pillow）：

```python
from PIL import Image, ImageDraw

def make_banner(width, height, base_color=(90, 5, 16)):
    img = Image.new('RGB', (width, height), base_color)
    draw = ImageDraw.Draw(img)
    line_color = (60, 3, 10)
    spacing = 30
    for x in range(-height, width + height, spacing):
        draw.line([(x, 0), (x + height + 20, height)], fill=line_color, width=2)
    return img
```

### 素材语义匹配要求

每镜素材必须与文案内容**语义直接相关**，不能间接关联或牵强对应：

| 文案内容 | 正确素材 | 错误素材（禁止） |
|----------|----------|------------------|
| "竹竿效应" | 竹竿/柱状递增动画 | 随意的通用柱状图 |
| "光纤光棒" | 光纤生产线/光纤拉丝工艺 | 金属拉丝图片/数据中心机房 |

| "城市夜景" | 纯净城市天际线 | 带"SHANGHAI $600 BILLION"水印的城市排名视频 |

判断标准：看到画面后，观众能否直觉理解它与文案的关联。需要牵强解释就不合格。

### 素材消音规则

平台实拍素材（B站/YouTube/抖音）下载后**必须自动消音**，只保留TTS语音轨：

```bash
# 下载后立即消音（去掉原始音频轨）
ffmpeg -y -i raw_clip.mp4 -an -c:v copy raw_clip_silent.mp4
```

- 所有平台实拍素材在进入合成前必须确认无音频轨（`ffprobe`检查`Stream`中不含`Audio`）
- 数据可视化动画和AI生成素材本身无音频，无需消音
- 最终视频音轨仅来自TTS，不混入任何素材原声

## 分步审查机制（每个Phase完成后强制执行）

每个Phase完成后，**必须截屏展示给用户检查**，用户确认通过后才进入下一个Phase：

### 审查流程

```
Phase完成 → 截屏/截帧展示 → 用户检查 → 用户确认通过
```

### 各Phase审查内容

| Phase | 审查方式 | 用户检查项 |
|-------|---------|-----------|
| Phase 0 分镜方案 | 表格展示 | 分镜数、时长、素材类型、关键词是否合理 |
| Phase 1 TTS | 播放音频或时长列表 | 音质、语速、时长是否匹配 |

### ⚠️ 禁止跳过审查

- **禁止**完成一个Phase后直接进入下一个Phase而不展示给用户
- **禁止**仅在文字描述结果而不截屏/截帧展示实际画面
- **禁止**在用户未确认通过时继续推进
- 每次进入新Phase前**必须重新查阅SKILL.md对应章节**，避免遗漏规格要求

## 素材类型优先级

1. **平台实拍素材**（B站/YouTube/抖音） — 先找抖音→B站→YouTube，搜索→预筛选→下载→截帧验收
2. **素材网站**（Pexels/Coverr/Pixabay） — 无水印直接可用，作为平台实拍的补充
3. **AI生成视频** — 优先尝试AI生成动态视频；若模型不具备能力，备选AI图片+FFmpeg运镜
4. **数据可视化动画** — Pillow/matplotlib生成的动画（柱状图、环形图、数字递增等）
5. **禁止**：PPT截图、K线截图、带水印的任何素材

### 素材占比方案（30句脚本参考）

| 素材类型 | 段数 | 占比 | 说明 |
|----------|------|------|------|

| AI生成视频 | 11 | 35% | 含8段数据可视化动画 + 3段AI生成动态视频 |
| 素材网站 | 9 | 30% | Pexels / Coverr / Pixabay |
| 合计 | 30 | 100% | |

> 占比可根据脚本实际内容调整，但平台实拍+素材网站合计不低于50%，AI生成+数据可视化不高于50%。
>
> **AI降级规则**：当AI视频生成不可用（只能用AI图片+FFmpeg运镜时），AI生成段数减半（35%→约18%），减少的段数改用素材网站素材填充（30%→约47%）。调整后占比参考：

> | 素材网站 | 14-15 | ~47% | 补充AI降级减少的部分 |
> | 合计 | 30 | 100% | |

### 素材横屏优先原则（核心规则）

**所有平台实拍素材和素材网站素材，优先截取横屏（16:9）画面。**

原因：竖版视频的素材净区是 1080×768（接近 4:3），横屏素材裁剪到 1080×768 时损失小、构图完整；竖屏素材（9:16）裁剪到 1080×768 需要裁掉上下大量内容，容易丢失主体信息。

| 素材方向 | 原始比例 | 裁剪到1080×768 | 效果 |
|----------|----------|----------------|------|
| 横屏 ✅ | 16:9 (1920×1080) | 上下加黑边或轻微裁剪 | 构图完整，主体居中 |
| 竖屏 ❌ | 9:16 (1080×1920) | 需裁掉上下各576px | 损失大量画面，容易裁掉主体 |

**搜索时优先筛选横屏视频**：
- 抖音/B站/YouTube 搜索时，优先选择横屏拍摄的视频（纪录片、航拍、工厂实拍等）
- 排除竖屏拍摄的短视频、口播类、手机拍摄类
- Pexels/Coverr 搜索时，在筛选条件中选择 Landscape / Horizontal 方向

**例外情况**：当横屏素材实在搜不到，且该镜内容必须用实拍画面时，才退而求其次用竖屏素材，但需额外裁剪处理确保主体不丢失。

### 素材时长要求（核心规则）

**截取素材必须先取10-30秒的连续片段，再按TTS时长裁剪：**

1. 从源视频截取一段 **10-30秒** 的连续流畅画面
2. 从这段素材中按TTS实际时长（3-8秒）做最终裁剪
3. **禁止**直接截取只有3秒的片段（画面来不及展开，生硬）
4. **禁止**使用截屏图片拼接成一卡一卡的伪视频
5. 素材画面必须**丝滑流畅**：有自然运动感，不能是静止画面或跳帧

```bash
# 正确：先截取15秒流畅片段，再按TTS时长裁剪
ffmpeg -i source.mp4 -ss 00:01:20 -t 15 -an raw_clip.mp4
ffmpeg -i raw_clip.mp4 -ss 0 -t 5 -vf "crop=...,scale=1080:768" -an final_clip.mp4

# ❌ 错误：直接截3秒
# ffmpeg -i source.mp4 -ss 00:01:20 -t 3 output.mp4  ← 禁止
```

### 同一长视频多段截取规则

同一个长视频可以在不同进度条截取多个不同画面用于不同分镜，但：
- 每个截取段的画面内容必须不同
- 同一个长视频最多截2-3段不同进度
- 每段仍需截取10-30秒流畅片段再裁剪
- **绝不重复用同一几秒画面贴到多个分镜**

## 平台素材采集详细流程（搜索→下载→清洗）

### 总流程

```
确定关键词 → 打开平台搜索 → 模拟人工浏览预筛选 → 下载 → 截取10-30秒片段 → 清洗裁剪 → 验收
```

### 平台优先级

1. **抖音** — 素材丰富、画面动感强，优先搜索
2. **B站** — 纪录片/科普类素材多，需处理cookies
3. **YouTube** — 英文搜索覆盖广，需代理+cookies（登录态，下载受限时必须）
4. **素材网站**（Pexels/Coverr/Pixabay） — 平台搜不到时的补充

### 抖音素材采集流程

1. 用Playwright打开 `https://www.douyin.com/search/{关键词}`
2. 在搜索结果页浏览视频列表，**模拟人工操作**：
   - 逐个hover视频缩略图，查看预览画面
   - 判断画面是否无人脸、无字幕、无水印、无字体、与文案匹配
   - 排除明显不合适的（UP主口播类、幻灯片类）
3. 点击进入候选视频，实际播放5-10秒确认画面内容
4. **获取视频真实URL**（二选一）：

   **方法A：转发复制链接 + evaluate提取（推荐）**
   - 在视频播放页点击右侧"转发/分享"按钮
   - 在弹出的分享面板点击"复制链接"
   - 用PowerShell `Get-Clipboard` 读取剪贴板，提取短链接（如 `https://v.douyin.com/xxxxx/`）
   - 用Playwright `browser_navigate` 打开短链接，跳转到视频详情页
   - 用 `browser_evaluate` 执行JS提取video标签的 `currentSrc`：
   ```javascript
   () => {
     const videos = document.querySelectorAll('video');
     const srcs = [];
     videos.forEach(v => srcs.push({src: v.src, currentSrc: v.currentSrc}));
     return JSON.stringify(srcs);
   }
   ```
   - 从返回JSON中取 `currentSrc` 字段（douyinvod.com域名的视频直链）
   - 用ffmpeg直接下载（带 `-t` 参数限制时长）：
   ```bash
   ffmpeg -y -i "{currentSrc_URL}" -t 30 -c:v libx264 -c:a aac raw_douyin_{id}.mp4
   ```

   **方法B：yt-dlp下载（备选，需cookies）**
   - 先用Playwright `browser_cookies` 工具获取抖音cookies，转为Netscape格式保存为 `douyin_cookies.txt`
   - yt-dlp可能报 "Fresh cookies needed" 错误，此时切换到方法A
   ```bash
   yt-dlp --cookies douyin_cookies.txt -f "best[height<=720]" -o "raw_douyin_%(id)s.mp4" "https://www.douyin.com/video/xxxxx"
   ```

5. 用FFmpeg截取10-30秒流畅片段，再按TTS时长裁剪到1080×768

### B站素材采集流程

1. 用Playwright打开 `https://search.bilibili.com/all?keyword={关键词}`
2. 模拟人工浏览搜索结果：
   - 关注视频封面和标题，排除口播类/教学类
   - 优先选择纪录片/工厂实拍/航拍类
3. 点击进入候选视频，**处理B站cookies**：
   - 首次访问B站会弹出登录弹窗，需关闭弹窗或处理cookies
   - 用Playwright的`browser_cookies`工具获取并保存cookies
   - 后续请求带上cookies避免被拦截
4. 实际播放视频5-10秒，确认画面干净（无人脸/字幕/水印）
5. 用yt-dlp下载（带cookies）：
```bash
yt-dlp --cookies bilibili_cookies.txt -o "raw_bili_%(id)s.mp4" "https://www.bilibili.com/video/BVxxxxx"
```
6. 截取10-30秒片段 → 裁剪字幕区 → 缩放1080×768

### YouTube素材采集流程

1. 用Playwright打开 `https://www.youtube.com/results?search_query={关键词}`
2. 模拟人工浏览：
   - 排除有人脸口播的频道视频
   - 优先选择与文案语义匹配的场景类型（工业/科技/自然/城市/金融/历史/教育等，按文案实际内容选择）
3. 播放确认画面内容
4. **Cookies获取与更新**（必须，YouTube下载受限需登录态）：
   - 用Playwright打开YouTube，确认已登录（右上角显示用户头像而非"登录"按钮）
   - 若未登录：点击右上角"登录"→用户完成Google账号登录→登录成功后继续
   - 用 `browser_cookies` 工具获取 `youtube.com` 域cookies（JSON格式）
   - 转为Netscape格式保存为 `youtube_cookies.txt`（注意：UTF-8无BOM，否则yt-dlp报错）
   ```python
   # JSON cookies转Netscape格式（无BOM）
   import json
   with open("cookies_youtube.com.json") as f:
       cookies = json.load(f)
   lines = ["# Netscape HTTP Cookie File"]
   for c in cookies:
       domain = c["domain"]
       flag = "TRUE" if domain.startswith(".") else "FALSE"
       httponly = "TRUE" if c.get("httpOnly") else "FALSE"
       expiry = int(c.get("expires_epoch", 9999999999)) if c.get("expires") else 9999999999
       lines.append(f"{domain}\t{flag}\t{c['path']}\t{httponly}\t{expiry}\t{c['name']}\t{c['value']}")
   with open("youtube_cookies.txt", "w", encoding="utf-8") as f:
       f.write("\n".join(lines) + "\n")
   ```
   - **Cookies过期检测**：下载时报 `Sign in to confirm you're not a bot` 或 `ERROR: unable to download API data` → cookies已过期，重新走上面流程获取
5. 用yt-dlp下载（带cookies）：
   ```bash
   # 短视频直接下载
   yt-dlp --cookies youtube_cookies.txt -f "best[height<=720]" --merge-output-format mp4 -o "raw_yt_%(id)s.mp4" "https://www.youtube.com/watch?v=xxxxx"

   # 长视频只下载所需片段（避免下载1-2小时整片，用 --download-sections）
   yt-dlp --cookies youtube_cookies.txt -f "best[height<=720]" --merge-output-format mp4 --download-sections "*00:30-00:50" -o "raw_yt_%(id)s.mp4" "https://www.youtube.com/watch?v=xxxxx"
   ```
   - `--download-sections "*起-止"` 格式：`*MM:SS-MM:SS`，只下载指定时间段，大幅节省时间
6. 截取10-30秒片段 → 裁剪 → 缩放1080×768

## AI生成素材要求（优先视频，备选图片+运镜）

### 优先方案：AI生成动态视频

AI生成素材**优先尝试生成动态视频**，而非静态图片。可用工具按优先级：

1. **WorkRally AI生视频**（需API Key，若已配置则优先使用）
2. **其他AI视频生成工具**（如有可用MCP工具）

若AI视频生成工具可用：
- 生成5-8秒动态视频，直接用于素材区
- 提示词：无文字、无人脸、与文案语义匹配的场景动态画面（场景类型由文案实际内容决定，如科技/工业/自然/城市/金融/历史/教育等，不限定某一类型）
- 生成后仍需验收12项检查

### 备选方案：AI生成图片 + FFmpeg运镜

**当AI视频生成不可用时**（如WorkRally API Key未配置），使用 `image_generate` 生成静态图片，再通过FFmpeg zoompan做运镜模拟动态效果。**不能直接贴静态图。**

```bash
# zoompan 缓慢放大
ffmpeg -y -i input.jpg -vf "scale=4000:-1,zoompan=z='min(zoom+0.0015,1.5)':d=150:s=1080x768:x='iw/2-(iw/zoom/2)':y='ih/2-(ih/zoom/2)':fps=30" -t 5 output.mp4

# zoompan 左右平移
ffmpeg -y -i input.jpg -vf "scale=4000:-1,zoompan=z=1.2:d=150:s=1080x768:x='if(lte(on,75),0,(iw-1080*1.2)*(on-75)/75)':y='ih/2-(ih/zoom/2)':fps=30" -t 5 output.mp4

# zoompan 上下平移
ffmpeg -y -i input.jpg -vf "scale=-1:4000,zoompan=z=1.2:d=150:s=1080x768:x='iw/2-(iw/zoom/2)':y='if(lte(on,75),0,(ih-768*1.2)*(on-75)/75)':fps=30" -t 5 output.mp4
```

### 运镜类型选择

| 转场运镜 | 适用场景 | FFmpeg参数 |
|----------|----------|------------|
| zoom-in | 强调细节、聚焦主体 | `zoom+0.0015`缓慢放大 |
| zoom-out | 展示全貌、拉开视野 | `zoom-0.0015`缓慢缩小 |
| pan-left | 展示横向场景（产线、流水线） | x从右到左移动 |
| pan-right | 展示横向场景（反向） | x从左到右移动 |
| tilt-up | 展示纵向场景（高楼、设备） | y从下到上移动 |
| tilt-down | 展示纵向场景（反向） | y从上到下移动 |
| static | 数据可视化动画自带动效 | 不加zoompan |

- 运镜类型根据画面内容和分镜方案中的转场运镜字段选择
- 持续时间匹配TTS时长
- 效果必须丝滑，不能生硬跳帧
- AI图片提示词：无文字、无人脸、与文案语义匹配的场景（场景类型由文案实际内容决定，不限定某一类型）

## 数据可视化动画

### ⚠️ 数据可视化触发条件（严格限定，违反即重做）

数据可视化**仅限以下三种情况触发**，其余一律禁止：

| # | 触发条件 | 示例 | 说明 |
|---|---------|------|------|
| T1 | 句子含**具体数值** | "涨价40%"、"192GB"、"从25%到45%" | 有明确数字才用数据可视化 |
| T2 | 句子含**明确比喻词** | "竹竿效应"、"缩圈"、"木桶效应" | 比喻词有对应图形表达 |
| T3 | 句子是**数值对比** | "从25%到45%"、"去年3亿今年8亿" | 两个数值之间的对比 |

**以下情况禁止使用数据可视化**，必须用实景画面或AI生成画面：

| 禁止类型 | 示例 | 正确做法 |
|---------|------|---------|
| 叙述句 | "AI基建趋势行情玩法变了" | 实景/AI画面（如芯片工厂、市场交易场景） |
| 观点句 | "去年是AI必涨" | 实景/AI画面（如牛市行情氛围画面） |
| 判断句 | "今年资金只认业绩验证" | 实景/AI画面（如财报、审计、资金流向场景） |
| 趋势句 | "资金从芯片转向MLCC" | 实景/AI画面（如产业链转移、工厂切换） |

**判断流程**：拿到一句文案 → 是否含具体数值？→ 是否含明确比喻词？→ 是否是数值对比？→ 三个都否 → **禁止数据可视化，用实景或AI画面**。

当文案满足触发条件时，**必须生成MG图形动画**，不能用静态图或无关AI图替代。

### ⚠️ 数据可视化铁律（代码级约束，违反即重做）

以下规则**不可变通**，违反任何一条必须推翻重做：

| # | 铁律 | 违反后果 |
|---|------|---------|
| D1 | **全中文**：画面所有文字必须中文，禁止英文（"old rule"/"narrowing"等一律禁止） | 重做 |
| D2 | **禁止纯文字动画**：不能只有文字淡入淡出/位移，必须有**图形元素**（柱状图/环形图/箭头/圆圈/进度条等）在动 | 重做 |
| D3 | **图形占比≥70%**：画面中图形动画元素面积≥70%，文字标注≤30% | 重做 |
| D4 | **必须有运动**：柱状图必须有增长动画、圆圈必须有缩放动画、箭头必须有移动动画，不能静态 | 重做 |
| D5 | **禁止HTML静态页**：不能做成静态HTML截图，必须是逐帧动画（Pillow帧序列或HTML Canvas+录屏） | 重做 |
| D6 | **配色统一（禁绿色）**：背景暗红渐变+斜线纹理，金色#e8c860标题，钢蓝#3264dc正向数据，橙红#dc5032负向数据，白色#fff文字。**全局禁止绿色**（对标Bloomberg/Wind金融终端配色） | 重做 |
| D7 | **分辨率1080x768**：输出视频必须1080x768，匹配素材区 | 重做 |
| D8 | **时长匹配TTS**：动画时长=该镜TTS时长，误差≤0.5秒 | 重做 |

### ⚠️ 数据可视化排版规范（金融数据动态看板标准，必须逐条遵守）

以下规范文件 `data_viz_spec.md` 是数据可视化的**唯一排版标准**，制作时必须逐条执行：

**画布分区系统（零文字重叠的根基）：**
- Z1标题区(y:0~90) / Z2主图形区(y:90~560,x:80~1000) / Z3数据标签区(y:560~640) / Z4轴标签区(y:640~720) / Z5脚注区(y:720~768)
- **铁律D1**：任何元素不得跨越分区边界
- **铁律D2**：画布四边40px安全区，有效绘图区x∈[40,1040],y∈[40,728]

**文字防碰撞（彻底解决重叠）：**
- **铁律D3**：文字与图形物理隔离，文字只在所属分区
- **铁律D4**：柱状图数值标签固定在柱子底部下方（y=580），禁止跟随柱顶
- **铁律D5**：饼图百分比写在饼图正下方Z3区，不在圆心；图例在右侧或下方
- **铁律D6**：每帧渲染后碰撞检测，碰撞则移文字到安全分区

**排版层级（字号不乱）：**
- L1标题=42px金色粗体 / L2数据=56~72px金色或白色粗体 / L3标签=28~32px白色常规 / L4脚注=22px灰色
- **铁律D7**：禁止范围外字号，禁止缩字号适配

**配色（禁绿色，金融看板风）：**
- 金色(232,200,96)=标题强调 / 钢蓝(50,100,220)=正向增长 / 橙红(220,80,50)=负向警示 / 浅蓝(100,150,255)=次要对比 / 白色=普通文字
- **铁律D10**：全局禁止绿色RGB，对标Bloomberg/Wind终端配色

**动态交互质感（像After Effects做出来的效果）：**
- **铁律D12-DYN**：入场用ease_out_back弹性缓动，标签用alpha淡入，柱状图背景画浅色参考线，整体视觉要像AE制作的动态看板一样有节奏感
- **铁律D13**：禁止线性动画，必须用ease_out/ease_out_back/ease_in_out缓动函数
- 当前活跃元素加金色描边高亮，非活跃元素暗化

**禁止清单（铁律D14）：**

```python
import subprocess, json, os

def check_data_viz(video_path, expected_dur, expected_w=1080, expected_h=768):
    """检查数据可视化动画是否合格"""
    issues = []
    
    # 1. 分辨率检查
    r = subprocess.run(["ffprobe", "-v", "error", "-select_streams", "v:0",
        "-show_entries", "stream=width,height,duration", "-of", "json", video_path],
        capture_output=True, text=True)
    info = json.loads(r.stdout)["streams"][0]
    w, h = int(info["width"]), int(info["height"])
    if w != expected_w or h != expected_h:
        issues.append(f"D7: 分辨率{w}x{h} != {expected_w}x{expected_h}")
    
    # 2. 时长检查
    dur = float(info["duration"])
    if abs(dur - expected_dur) > 0.5:
        issues.append(f"D8: 时长{dur:.1f}s != {expected_dur}s")
    
    # 3. 截帧检查是否有运动（对比第1帧和中间帧的文件大小差异）
    subprocess.run(["ffmpeg", "-y", "-ss", "0.5", "-i", video_path, "-frames:v", "1", "check_f1.jpg"],

        capture_output=True)
    s1 = os.path.getsize("check_f1.jpg")
    s2 = os.path.getsize("check_f2.jpg")
    if abs(s1 - s2) / max(s1, s2) < 0.05:
```

```python
import subprocess

def make_bar_chart_animation(title, labels, values, output_path, duration=5, fps=30):
    frames = []
    total_frames = duration * fps
    max_val = max(values)

    for i in range(total_frames):
        img = Image.new('RGB', (1080, 768), '#0a0a0a')
        draw = ImageDraw.Draw(img)
        progress = min(i / (total_frames * 0.7), 1.0)

        font_title = ImageFont.truetype('msyhbd.ttc', 36)
        draw.text((540, 50), title, fill='#e8c860', font=font_title, anchor='mm')

        bar_width = 120
        bar_gap = 40
        start_x = 200
        base_y = 650
        max_bar_h = 500

        for j, (label, value) in enumerate(zip(labels, values)):
            x = start_x + j * (bar_width + bar_gap)
            h = int(max_bar_h * (value / max_val) * progress)
            color = '#e8c860' if j == 0 else '#5a0510'
            draw.rectangle([x, base_y - h, x + bar_width, base_y], fill=color)
            font_label = ImageFont.truetype('msyh.ttc', 24)
            draw.text((x + bar_width//2, base_y + 20), label,
                     fill='#fff', font=font_label, anchor='mm')
            if progress > 0.5:
                font_val = ImageFont.truetype('msyhbd.ttc', 28)
                draw.text((x + bar_width//2, base_y - h - 20),
                         f'{int(value * progress)}%',
                         fill='#e8c860', font=font_val, anchor='mm')
        frames.append(img)

    frames[0].save('anim.gif', save_all=True, append_images=frames[1:],
                   duration=33, loop=0)
    subprocess.run(['ffmpeg', '-y', '-i', 'anim.gif', '-vf', 'fps=30', output_path])
```

## 长句分幕规则（核心）

当一句文案过长（超过20字或超过文案条宽度）时：

1. **不拆TTS** — 语音仍是一整句连续播放，TTS对整句合成
2. **拆字幕** — 将文案按语义拆成2-4个子句，子句在文案条上轮播
3. **素材视频不变** — 同一镜的素材视频连续播放，不切换
4. **字幕轮播** — 按子句时间点轮流切换文案条文字

### ⚠️ 错误做法（禁止）

把长句拆成多个独立子句，每个子句单独做TTS，导致语音断续、不自然。

### 正确做法示例

```
脚本原句："台积电全线涨价5%到10%，排队等产能排到明年。"

✅ 正确：
  - TTS：整句合成一次 → "台积电全线涨价5%到10%，排队等产能排到明年。"（约6秒连续语音）
  - 字幕子句1（0-3s）："台积电全线涨价5%到10%，"
  - 字幕子句2（3-6s）："排队等产能排到明年。"
  - 素材视频：同一段6秒工厂画面连续播放
  - 文案条：0-3s显示子句1，3-6s显示子句2

❌ 错误：
  - TTS_0："台积电全线涨价5%到10%，"（3秒，独立合成）
  - TTS_1："排队等产能排到明年。"（3秒，独立合成）
  - 两个独立的3秒视频段，语音不连续
```

### 子句时间分配

按字数比例分配各子句在文案条上的显示时间。需要分析TTS音频中子句的自然停顿点。

## 素材验收12项检查（每段素材下载/生成后强制执行）

用FFmpeg按1秒间隔截帧，**全帧验证**（不是抽帧），每帧逐项检查：

| # | 检查项 | 合格标准 | 不合格处理 |
|---|--------|---------|-----------|
| 1 | 画面方向 | 裁切后能填满素材区1080×768 | 丢弃换源 |
| 2 | 画面内容相关性 | 与文案意象直接相关 | 丢弃换源 |
| 3 | 动态视频 | 有物体移动/镜头移动/场景变化 | 丢弃换源 |
| 4 | 无水印/无logo | 无任何水印、logo、台标、UI弹窗 | 裁剪或模糊处理；无法处理则丢弃 |
| 5 | **无字幕/无文字/无字体** | 画面无任何字幕、文字、弹幕、印刷字体（零容忍） | 丢弃换源 |
| 6 | **无人脸** | 无UP主口播、无博主大头、无半身半脸（零容忍） | 丢弃换源 |
| 7 | 画面审美 | 构图完整、主体居中、清晰不模糊、不丑 | 丢弃换源 |
| 8 | 分辨率 | 裁剪后有效区域≥720×768px | 丢弃换源 |
| 9 | 时长 | 可截取≥6秒有效片段 | 丢弃换源 |
| 10 | **无K线图截取** | 不能截取真实K线图/走势图 | 丢弃换源 |
| 11 | **无PPT/无录屏** | 不能截取PPT/手机录屏/电脑录屏 | 丢弃换源 |
| 12 | **无无关画面** | 必须与当前分镜文案直接相关 | 丢弃换源 |

**任何一项不合格 → 整段丢弃，不裁剪凑合，不模糊凑合。**

### 素材反馈机制（候选素材截帧对比图）

每段平台实拍素材在验收前，**必须生成候选截帧对比图供用户确认**，不能仅文字描述：

1. 每个候选视频用FFmpeg截取4帧（间隔均匀），拼成2×2网格图

3. 同时对所有候选视频的截帧图并列展示（如3个候选→3张对比图）
4. 用 `image` 工具逐张分析，附上6项判断结果：
   - 无人脸 / 无字幕文字字体 / 无水印logo / 无K线图表 / 画面与文案匹配 / 画面动感
5. 将分析结果和截帧图一起呈现给用户，由用户确认选用哪个候选
6. 用户确认后才进入裁剪清洗和最终验收

```
候选C（YouTube）→ 截4帧 → 拼网格图 → image分析 → 标注6项结果
```

```bash
      x=(w-text_w)/2:y=(h-text_h)/2:
      enable='between(t,3,6)'[sub2];
    [top][vm][sub1][sub2][bot]vstack=inputs=5[vo]
  " \
  -map "[vo]" -map 1:a \
  -c:v libx264 -c:a aac -shortest seg_NN.mp4
```

## Phase 5：全帧验证（强制，零容忍）

**最终视频每2秒截帧，全片至少110帧。**

```bash
# 对最终视频每隔2秒抽帧
for t in $(seq 0 2 $total_duration); do
    ffmpeg -y -ss $t -i final.mp4 -frames:v 1 "frame_$(printf '%03d' $t).jpg"
done
```

### 最终视频逐段验证项

| 验证项 | 方法 | 合格标准 |
|--------|------|---------|
| 标题区 | 每段截1帧 | 暗红底+金色粗体标题（不斜体），字号可见不溢出 |
| 标题斜体 | 每段截1帧 | 标题文字必须是斜体 |
| 标题自适应 | 长标题截1帧 | 自动缩小字号适应宽度，不溢出 |
| 素材区内容 | 每段截3帧（开头/中段/结尾） | 与文案匹配，无水印/字幕/人脸/K线/PPT |
| 文案条 | 每段截1帧 | 暗红底+白字，文字可读，只显示当前一句 |
| 文案逐句切换 | 逐段检查 | 字幕只显示当前一句，不是全部堆一起 |
| 长句分幕 | 检查长句段 | 长句已拆子句，素材不变但字幕轮播 |
| 下贴片 | 每段截1帧 | 暗红底+斜线纹理，无标题文字 |
| 贴片颜色 | 每段截1帧 | 统一暗红#5a0510，无色差，有斜线纹理 |
| AI生成素材 | 每段截3帧 | 有明显动态运镜效果，不是静止画面 |
| 音画同步 | 逐段检查 | 语音与字幕对应，时间点匹配 |

发现问题后：
1. 记录问题镜号和帧时间
2. 替换该镜素材
3. 重新合成该镜
4. 重新验证

## Phase 6：最终合成

1. 所有 `seg_NN/final.mp4` 按顺序concat拼接
2. TTS音轨按顺序合并（每镜一段完整句音频）
3. 音视频合成

```bash
# 视频拼接
ffmpeg -y -f concat -safe 0 -i concat.txt -c copy final_video.mp4

# 音频拼接
ffmpeg -y -f concat -safe 0 -i audio_concat.txt -c copy full_audio.mp3

# 音视频合并
ffmpeg -y -i final_video.mp4 -i full_audio.mp3 -c:v copy -c:a aac -shortest final.mp4
```
